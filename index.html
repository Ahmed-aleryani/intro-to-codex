03

The AI Development Cycle

Two loops, not one

Part 3

Two Loops Working Together

When building with AI, there are two cycles happening:

The Outer Loop

You drive this. The big-picture cycle between you and the AI.

Plan → Execute → Review → repeat

The Inner Loop

The AI drives this. It runs inside the Execute phase.

Write → Test → Fix → repeat

Part 3

The Outer Loop (You Drive This)

1. Plan

Describe what you want. The AI creates a step-by-step plan. You review and refine.

2. Execute

Approve the plan. The AI builds it. Let it work — don't interrupt.

3. Review

Check the output. Good? Next task. Not right? Back to planning.

This is a continuous loop — not a one-time process. Every feature, every fix goes through its own Plan → Execute → Review cycle.
Part 3

The Inner Loop (AI Drives This)

During Execute, the AI runs its own internal cycle — writing, testing, finding problems, and fixing them before showing you the result.

Write code → Run tests → See errors → Fix & retry ↻
When you ask a chef to make dinner, they taste the sauce, adjust seasoning, taste again. You only see the final plate. The AI works the same way — it self-corrects behind the scenes.
Part 3

When Should You Intervene?

Let it work

  • It's making progress (even if slow)
  • It's running tests and fixing errors on its own
  • The task has clear success criteria

Step in

  • Stuck on the same error repeatedly
  • Output going in completely wrong direction
  • The original plan needs to change

You are the architect. The AI is the builder.
You decide what to build. The AI figures out how — including fixing its own mistakes.

04

Context

The AI's working memory

Part 4

Context — The AI's Working Memory

Everything the AI can "see" during a conversation — like its desk. Too little and it forgets; too much and quality drops.

Golden Rule: Keep each task focused on one clear thing.

Asking a contractor to fix your faucet? Show them the kitchen, not the blueprints for the entire building.

Practical Tips

  • Start fresh conversations for new features
  • Reference specific files instead of "look at everything"
  • Break big tasks into small ones — each gets its own cycle
  • Use agents.md for persistent context (up next!)

agents.md

Teaching AI your project's rules

Part 5

What is agents.md?

A text file at your project root — a rulebook for AI assistants. Used by 60,000+ projects.

Instead of explaining the same rules to your AI every session, write them down once.

Aug 2025 — OpenAI introduced the standard with Codex

Dec 2025 — Donated to the Linux Foundation (neutral governance)

Today — Backed by OpenAI, Google, GitHub, Cursor, and many others

Part 5

Which File Should You Use?

Using one AI tool?

Just use that tool's file. Claude Code → CLAUDE.md. Codex → AGENTS.md. Simple.

Using multiple AI tools?

Write one AGENTS.md and create shortcuts (symlinks) so all tools read it:

ln -s AGENTS.md CLAUDE.md
ln -s AGENTS.md GEMINI.md

A symlink is like a desktop shortcut — the file points to AGENTS.md, so you only maintain one file.

The most important section ↓

Part 5

Boundaries — The Three Tiers

Tell the AI what it's allowed to do:

ALWAYS DO (no permission needed)

  • Read and analyze files
  • Run linting and formatting
  • Write tests

ASK FIRST (needs your approval)

  • Install new packages
  • Create new files
  • Change the database

NEVER DO

  • Delete files without asking
  • Push code without review
  • Modify secrets or environment variables
  • Install unvetted, low-quality dependencies
Part 5

Best Practices

  • Keep it short — under 150 lines
  • Be specific — camelCase beats "follow best practices"
  • Show examples — code snippets beat descriptions
  • Start simple — commands + boundaries first
  • Iterate — AI makes a mistake? Add a rule
  • Only trusted deps — high stars, high downloads
